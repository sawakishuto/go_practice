# ドメインイベント・イベントソーシング・メッセージキュー（読み物）

[TRAINING.md](./TRAINING.md) **Phase 4** の手を動かす前後で読むと、用語とトレードオフがつながります。ここでは **実装を完成させない前提の概念整理** に留めます。

---

## 1. ドメインイベントとは

**すでにドメインの中で起きた事実**を表すデータ（構造体）です。

- **コマンド**（`Borrow` を試みる）… まだ結果が決まっていない。失敗しうる。
- **イベント**（`BookBorrowed`）… **ルールを通過したあと**にだけ成り立つ「起きたこと」の記録。

イベントは **過去形の名前**（`...Occurred` / `...Registered` / `...Borrowed`）にすると、コマンドと区別しやすいです。

---

## 2. ドメインイベントとインテグレーションイベント

| 種類 | 伝える相手のイメージ | 例 |
|------|----------------------|-----|
| **ドメインイベント** | **同じ境界づけられたコンテキスト**内のルール・集約 | `BookBorrowed`（本棚ドメイン） |
| **インテグレーションイベント** | **別システム・別コンテキスト**（検索、請求、通知） | `BookCatalogUpdated` をバスに載せる |

小さな学習リポジトリでは最初は **ドメインイベントだけ**でよいです。名前を **`book` パッケージに置く**と「ドメインの語彙」であることが明確になります。

---

## 3. 「イベント駆動」と「イベントソーシング」

| | イベント駆動（広い意味） | イベントソーシング（狭い意味） |
|---|--------------------------|--------------------------------|
| **保存する主役** | 現在の状態（行の UPDATE）でもよい | **イベント列（ストリーム）**を正とする |
| **状態の復元** | 最新行を読む | **イベントを先頭から適用（リプレイ）**して組み立てる |
| **難しさ** | 比較的低い | **競合・バージョン・スナップショット**など設計が増える |

Phase 4 では **両方に触れる**: まず **イベントを発行して購読者に渡す**（イベント駆動の最小）、のち **追記ストア + リプレイ**（ES の入り口）。

---

## 4. イベントソーシングの最小モデル

1. **ストリーム ID** … 1 冊の本なら `book:{id}` のように **集約インスタンスごと**に 1 本の列を持つイメージ。
2. **追記（append-only）** … 過去のイベントを **上書きしない**。訂正は **打ち消しイベント**や **補正イベント**で表す（本格では方針が分かれる）。
3. **バージョン / 楽観ロック** … 同時に 2 つのコマンドが来たとき **同じバージョンを前提に書くと失敗**させる、など。学習用では **単一 goroutine の係員**や **単一プロセス**で省略してもよい。
4. **リプレイ** … `events` を順に `Apply(event)` して **`Book` の状態**を再現する純関数にするとテストしやすい。
5. **スナップショット** … イベントが数万件になったら **ある時点の状態をキャッシュ**してリプレイ開始点を短くする（Phase 4 では任意）。

## 5. メッセージキュー（MQ）で押さえること

### 5.1 なぜキューを挟むか

- **処理の分離**（借りる処理と通知・分析を切り離す）
- **吸収**（ピーク時にメモリではなくキューに溜める）
- **再試行**（失敗したコンシューマがあとから再処理）

### 5.2 配信保証のざっくり

| 用語 | 意味 |
|------|------|
| **at-most-once** | 最大 1 回。届かないことがある。 |
| **at-least-once** | 最低 1 回。**二重配信がありうる**（冪等性が重要）。 |
| **exactly-once** | 厳密には難しいことが多く、**ビジネス上の exactly-once**（冪等キー + 重複排除）で近似する。 |

### 5.3 並び順

- **1 ストリーム内は順序を守りたい**（同一 `bookID` のイベント）が、**グローバル順は要らない**ことが多い。
- **パーティションキー**（例: `bookID`）で **同じキーは同じコンシューマ**に寄せると順序が保ちやすい（Kafka のパーティションのイメージ）。

### 5.4 冪等性（idempotency）

**同じイベントが 2 回届いても壊れない**ようにする。例:

- **イベント ID** を付け、処理済みセットに載せたらスキップ。
- **「借りた回数」を += 1 する**のではなく、**「借りた事実」を ID で一意に記録**する。

### 5.5 トランザクションアウトボックス（Transactional Outbox）

**DB にコミット**と **MQ に publish** を別々にすると、**片方だけ成功**しうる。

**アウトボックス**は:

1. ビジネスデータと **同じトランザクション**で「未送信イベント」を **テーブルに INSERT**。
2. **別プロセス／ポーラー**が未送信行を読み、MQ に送って **送信済みフラグ**を立てる。

こうすると **「DB に載った事実は必ずどこかで publish される」** 方向に寄せられる（厳密な保証はブローカー設定とも依存）。

Phase 4 の **インメモリ実装**ではアウトボックスまでは必須にしないが、**「なぜ本番で必要か」** を知っておくとよいです。

---

## 6. 実装でハマりやすい点

- **イベントに集約の不変条件を書きすぎる** … ルールは **エンティティ**、イベントは **結果の記録**に寄せる。
- **巨大な `interface{}` 一つ** … 型スイッチが地獄になる。**型安全な sealed interface**（メソッド `isEvent()` を小文字で実装する trick）や **Discriminated union 風の struct + Kind 列挙**で段階的に。
- **リプレイで副作用** … リプレイは **状態の再計算だけ**。メール送信は **リプレイしない投影**側へ。
